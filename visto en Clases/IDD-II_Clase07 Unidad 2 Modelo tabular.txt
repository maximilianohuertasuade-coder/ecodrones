Datos distribuidos y eventos masivos: modelo tabular y CQL 
Bases de datos de columna ancha 
Las bases de datos de columna ancha, también llamadas bases de datos tabulares, 
columnares extendidas o wide-column databases, son un tipo de base de datos NoSQL 
diseñado para almacenar grandes volúmenes de información distribuida. Aunque 
utilizan conceptos como tablas, filas y columnas, no deben confundirse con una base 
de datos relacional tradicional. 
En una base relacional, una tabla suele tener una estructura fija: todas las filas 
comparten las mismas columnas, existen relaciones entre tablas, se usan claves 
primarias y foráneas, y las consultas pueden combinar información mediante JOIN. En 
cambio, en una base de columna ancha, la estructura es más flexible y está orientada a 
distribuir datos eficientemente entre varios nodos. El material base describe este 
modelo como una estructura tabular en la que cada fila puede tener una configuración 
diferente de columnas, con capacidad para manejar muchas columnas y grandes 
volúmenes de datos . 
Este tipo de modelo aparece como respuesta a problemas donde una base relacional 
puede volverse costosa o poco eficiente: 
• Registro masivo de eventos.  
• Logs de aplicaciones.  
• Historial de actividad de usuarios.  
• Series temporales.

• Sensores IoT.  
• Trazabilidad de transacciones.  
• Monitoreo de sistemas.  
• Eventos de seguridad.  
• Clics, búsquedas, sesiones y comportamiento digital.

Por ejemplo, una plataforma digital puede generar millones de eventos por día: 
usuario_1001 inició sesión a las 08:10 
usuario_1001 realizó una búsqueda a las 08:12 
usuario_1001 agregó un producto al carrito a las 08:15 
usuario_1001 realizó un pago rechazado a las 08:18 
usuario_1001 intentó ingresar desde otro dispositivo a las 08:23 
Guardar estos datos en una tabla relacional tradicional no es imposible. Sin embargo, 
si el volumen crece demasiado y se necesita consultar rápidamente por usuario, fecha, 
tipo de evento o dispositivo, el diseño relacional puede requerir índices complejos, 
particionamiento, joins, optimizaciones y mantenimiento permanente. 
Las bases de columna ancha buscan resolver este problema desde otra lógica: 
distribuir los datos desde el diseño y organizarlos según las consultas que se necesitan 
responder. 
De la tabla relacional a la tabla orientada a consulta 
En SQL Server, el punto de partida habitual es el modelo de datos. Se identifican 
entidades, atributos y relaciones. Luego se normaliza el modelo para evitar 
redundancia. Por ejemplo, en un sistema de eventos se podrían crear tablas como: 
Usuarios 
Eventos 
TiposEvento 
Dispositivos 
Sesiones 
Después, mediante consultas SQL, se combinan esas tablas para obtener la 
información necesaria. 
En Cassandra, el razonamiento cambia. El punto de partida no es solamente “qué 
entidades existen”, sino principalmente: ¿Qué consulta necesito responder rápido? 
Esto modifica el diseño. Una tabla en Cassandra no siempre representa una entidad del 
negocio. Muchas veces representa una consulta específica. 
Por ejemplo, si se necesita consultar los eventos de un usuario por día, podría existir 
una tabla llamada:

eventos_por_usuario_dia 
Si también se necesita consultar eventos por tipo y fecha, podría existir otra tabla: 
eventos_por_tipo_fecha 
Y si se necesita consultar eventos sospechosos por nivel de riesgo, podría existir otra:

El mismo evento puede repetirse en varias tablas. Esto, que en un modelo relacional 
parecería una mala práctica por redundancia, en Cassandra puede ser una decisión 
correcta de diseño. La desnormalización permite evitar joins y lecturas costosas. El 
material de modelado de Cassandra remarca precisamente que no se comienza con el 
modelo de datos, sino con el modelo de consulta, y que las tablas se crean para 
soportar las rutas de consulta más frecuentes . 
La idea central es: En Cassandra no se diseña una tabla para guardar “todo”. Se diseña 
una tabla para responder bien una pregunta. 
Cassandra: una base distribuida para grandes volúmenes 
Apache Cassandra es una base de datos NoSQL de columna ancha, distribuida y 
escalable horizontalmente. Está pensada para escenarios donde se necesita alta 
disponibilidad, tolerancia a fallos y manejo de grandes volúmenes de escritura y 
lectura. – CQL: Lenguaje de consulta de Casandra: es el lenguaje oficial de Apache 
Cassandra para interactuar con la base de datos. Su sintaxis es muy similar a SQL, 
permitiendo crear esquemas (DDL) y gestionar datos (DML) de forma sencilla, pero 
está optimizado para la arquitectura distribuida y NoSQL de Cassandra 
Cassandra se utiliza especialmente cuando los datos deben distribuirse entre varios 
nodos. Un nodo es una máquina del clúster que almacena parte de los datos. El clúster 
completo está formado por varios nodos que trabajan de manera coordinada. A 
diferencia de arquitecturas con un servidor central principal, Cassandra trabaja con 
una arquitectura distribuida donde los nodos se comunican entre sí y cualquiera de 
ellos puede recibir una operación. El material de referencia indica que los clientes 
pueden conectarse a cualquier nodo y que ese nodo actúa como coordinador de la 
operación frente al resto de los nodos involucrados . 
Esto permite comprender una de sus principales fortalezas: la escalabilidad horizontal. 
Si el volumen de datos o consultas crece, la solución no depende únicamente de hacer 
más potente un servidor, sino de agregar más nodos al clúster. 
Cassandra resulta especialmente adecuada cuando se combinan estas condiciones: 
• Mucho volumen de datos.  
• Muchas escrituras.  
• Datos distribuidos.

No es la opción ideal cuando se necesitan muchas consultas libres, joins complejos,

El cambio de mentalidad: de normalizar a consultar 
El cambio más importante al trabajar con Cassandra no es la sintaxis, sino la forma de 
pensar el diseño. 
En una base relacional, el razonamiento habitual es: 
Entidades → Relaciones → Normalización → Consultas 
En Cassandra, el razonamiento recomendado es: 
Consultas → Clave de partición → Clave de ordenamiento → Tabla optimizada 
Esto significa que antes de crear tablas hay que definir las preguntas que el sistema 
debe responder. 
Por ejemplo: 
• ¿Cuáles fueron los últimos eventos de un usuario en un día?  
• ¿Cuáles fueron los eventos sospechosos de una fecha?  
• ¿Cuántos intentos fallidos tuvo un usuario en una ventana de tiempo?  
• ¿Qué transacciones se registraron para un cliente?  
• ¿Qué eventos ocurrieron en determinado dispositivo?  
• ¿Qué errores generó una aplicación durante la última hora?  
Cada una de estas preguntas puede requerir una tabla distinta. 
Un error común sería intentar crear una única tabla general: 
eventos 
Con columnas como: 
evento_id, usuario_id, fecha, tipo_evento, ip, dispositivo, riesgo, detalle

Y luego pretender consultar libremente por cualquier columna, como se haría en SQL. 
En Cassandra esto no funciona bien, porque las consultas deben estar alineadas con la 
clave primaria de la tabla. Por eso, el diseño debe anticipar el acceso a los datos. 
La frase clave es: Primero se diseña la consulta; después se diseña la tabla. 
Clave primaria, clave de partición y clave de clustering 
La clave primaria en Cassandra cumple un rol fundamental. No solo identifica registros. 
También define cómo se distribuyen y ordenan los datos. 
Una clave primaria puede tener dos componentes: 
Clave de partición: Determina en qué partición se almacenan los datos. En términos 
simples, indica cómo se agrupan físicamente los datos y cómo se distribuyen entre los 
nodos. 
Clave de clustering: Determina cómo se ordenan los datos dentro de una misma 
partición. 
Row key es el equivalente a la clave primaria, compuesta por clave de particionado y 
clave de agrupamiento: la primera permite almacenar juntas las filas con el mismo 
valor de partición, mientras la segunda determina el orden físico de almacenamiento 
dentro de esa partición . 
Ejemplo conceptual: 
CREATE TABLE eventos_por_usuario_dia ( 
    usuario_id text, 
    fecha date, 
    evento_ts timestamp, 
    evento_id uuid, 
    tipo_evento text, 
    ip text, 
    dispositivo text, 
    resultado text, 
    PRIMARY KEY ((usuario_id, fecha), evento_ts, evento_id) 
) WITH CLUSTERING ORDER BY (evento_ts DESC); 
En esta tabla: 
(usuario_id, fecha) = clave de partición 
evento_ts, evento_id = claves de clustering 
La clave de partición agrupa todos los eventos de un usuario en una fecha 
determinada. La clave de clustering ordena esos eventos por momento de ocurrencia. 
El evento_id ayuda a garantizar unicidad cuando dos eventos pudieran tener el mismo 
timestamp.

FROM eventos_por_usuario_dia 
WHERE usuario_id = 'USR1001'

FROM eventos_por_usuario_dia

La columna tipo_evento no forma parte de la clave de partición. Por lo tanto, 
Cassandra no puede usarla como ruta principal de acceso eficiente. Para esa necesidad 
convendría otra tabla. 
Series temporales y eventos masivos 
Uno de los usos más frecuentes de Cassandra es el almacenamiento de eventos en el 
tiempo. Una serie temporal es una secuencia de datos registrados en distintos 
momentos. Cada registro tiene una marca de tiempo y representa algo que ocurrió. 
Ejemplos: 
Lecturas de temperatura de un sensor. 
Intentos de login de un usuario. 
Transacciones de una billetera virtual. 
Eventos de navegación en una aplicación. 
Cambios de estado de un pedido. 
Alertas de seguridad. 
Errores generados por un servidor. 
En estos casos, el tiempo no es un dato más: es parte central del diseño. 
Una mala decisión sería crear una partición demasiado grande. Por ejemplo: 
PRIMARY KEY (usuario_id, evento_ts) 
Este diseño agruparía todos los eventos históricos de un usuario en una única 
partición. Si el usuario genera millones de eventos, esa partición crecería demasiado. 
Una mejor alternativa suele ser dividir por períodos:

PRIMARY KEY ((usuario_id, fecha), evento_ts) 
De esta forma, los eventos se agrupan por usuario y día. También podría usarse 
usuario y mes, dispositivo y día, sensor y hora, comercio y fecha, según el volumen 
esperado. 
La decisión depende de la cantidad de datos. No es lo mismo registrar 10 eventos por 
usuario por día, que registrar: 500.000 eventos por dispositivo por hora 
En Cassandra, el diseño debe evitar dos extremos: 
• Particiones demasiado grandes, porque concentran muchos datos en un mismo 
grupo.  
• Particiones demasiado pequeñas o dispersas, porque obligan a buscar en 
demasiados lugares.  
El objetivo es lograr particiones razonables: suficientemente grandes para agrupar 
datos relacionados, pero no tan grandes como para afectar el rendimiento. 
Primer acercamiento a CQL 
CQL, Cassandra Query Language, es el lenguaje utilizado para crear y consultar 
estructuras en Cassandra. Su sintaxis se parece a SQL, pero su lógica no es la misma. 
CQL permite crear keyspaces, tablas, insertar datos, consultar, actualizar y eliminar 
registros. La referencia de CQL muestra comandos como CREATE KEYSPACE, CREATE 
TABLE, INSERT, UPDATE, DELETE, SELECT, ORDER BY, LIMIT y ALLOW FILTERING . 
Un CREATE TABLE básico podría ser: 
CREATE TABLE eventos_por_usuario_dia ( 
    usuario_id text, 
    fecha date, 
    evento_ts timestamp, 
    evento_id uuid, 
    tipo_evento text, 
    canal text, 
    ip text, 
    dispositivo text, 
    resultado text, 
    riesgo int, 
    detalle text, 
    PRIMARY KEY ((usuario_id, fecha), evento_ts, evento_id) 
) WITH CLUSTERING ORDER BY (evento_ts DESC); 
Este código no solo crea una tabla. También define la forma en que Cassandra 
organizará los datos.

La parte más importante es: PRIMARY KEY ((usuario_id, fecha), evento_ts, evento_id) 
La doble estructura indica: 
• Clave de partición: usuario_id + fecha 
• Claves de clustering: evento_ts + evento_id 
El orden descendente permite recuperar primero los eventos más recientes: 
WITH CLUSTERING ORDER BY (evento_ts DESC); 
Una consulta válida sería: 
SELECT tipo_evento, evento_ts, ip, dispositivo, resultado 
FROM eventos_por_usuario_dia 
WHERE usuario_id = 'USR1001' 
AND fecha = '2026-05-03' 
LIMIT 10; 
Esta consulta es válida porque usa la clave de partición completa. 
Una consulta problemática sería: 
SELECT * 
FROM eventos_por_usuario_dia 
WHERE riesgo >= 80; 
Aunque riesgo exista como columna, no fue elegida como clave de partición ni 
clustering. Por lo tanto, Cassandra no puede usarla como acceso principal eficiente. 
Para consultar eventos por riesgo, se debería pensar otra tabla, por ejemplo: 
CREATE TABLE eventos_por_riesgo_fecha ( 
    nivel_riesgo int, 
    fecha date, 
    evento_ts timestamp, 
    evento_id uuid, 
    usuario_id text, 
    tipo_evento text, 
    ip text, 
    detalle text, 
    PRIMARY KEY ((nivel_riesgo, fecha), evento_ts, evento_id) 
) WITH CLUSTERING ORDER BY (evento_ts DESC);

Cassandra no debe entenderse como una versión distribuida de SQL Server. Es otro 
modelo de diseño. 
Su fortaleza aparece cuando los datos son masivos, distribuidos, de alta escritura y con 
patrones de consulta conocidos. Su dificultad principal es que exige pensar antes qué 
se va a consultar. Una tabla mal diseñada puede obligar a lecturas costosas, uso 
indebido de filtros o duplicación desordenada de datos. 
El aprendizaje central puede resumirse así: 
En SQL se normaliza para representar correctamente el dominio. 
En Cassandra se desnormaliza para responder eficientemente consultas específicas. 
Y, especialmente para eventos masivos: 
La clave de partición define cómo se agrupan y distribuyen los eventos. 
La clave de clustering define cómo se ordenan dentro de cada grupo. 
El tiempo suele formar parte del diseño, no solo del dato. 
Una buena tabla Cassandra no es la que guarda todos los datos posibles, sino la que 
responde de manera eficiente una consulta importante.

Guía de laboratorio – Clase 7 
Cassandra, CQL y diseño orientado a consultas 
Objetivo del laboratorio 
El objetivo de esta práctica es comprender cómo se trabaja con Cassandra desde la 
consola cqlsh, utilizando comandos básicos de CQL y diseñando tablas orientadas a 
consultas. La actividad parte de una tabla simple para entender la sintaxis y luego 
avanza hacia el diseño correcto para eventos masivos, aplicando claves de partición, 
claves de clustering y consultas eficientes. 
La idea central de la práctica es la siguiente: 
En Cassandra no se diseña primero una tabla general para consultar cualquier cosa. 
Primero se define qué consulta se necesita responder. 
Luego se crea una tabla optimizada para esa consulta. 
El laboratorio toma como referencia la lógica de práctica con CREATE KEYSPACE, 
CREATE TABLE, INSERT, SELECT, consultas por clave primaria y diseño de tablas 
específicas para evitar ALLOW FILTERING, tal como aparece en la guía base de 
Cassandra .

2. Crear el keyspace de trabajo

6. Probar una consulta problemática 
Ahora intentar consultar por usuario: 
SELECT * 
FROM eventos 
WHERE usuario_id = 'USR1001'; 
Es esperable que Cassandra rechace la consulta o indique que no puede ejecutarla sin 
filtrado. Esto ocurre porque usuario_id no forma parte de la clave primaria. 
Análisis 
En SQL Server sería normal consultar: 
SELECT * 
FROM eventos 
WHERE usuario_id = 'USR1001'; 
Pero en Cassandra no alcanza con que la columna exista. La tabla debe estar diseñada 
para permitir esa consulta. 
Esta diferencia es central: CQL se parece a SQL en la sintaxis, pero Cassandra no 
permite consultar libremente por cualquier columna si la tabla no fue diseñada para 
eso.

    resultado text, 
    riesgo int, 
    PRIMARY KEY ((usuario_id, fecha), evento_ts, evento_id) 
) WITH CLUSTERING ORDER BY (evento_ts DESC); 
Explicación del diseño 
La clave primaria es: 
PRIMARY KEY ((usuario_id, fecha), evento_ts, evento_id) 
Esto significa: 
Clave de partición: usuario_id + fecha 
Clave de clustering: evento_ts + evento_id

La clave de partición agrupa los eventos de un mismo usuario en un mismo día. 
La clave de clustering ordena los eventos dentro de esa partición. 
El orden descendente permite consultar primero los eventos más recientes.

11. Probar una consulta incompleta 
Ejecutar: 
SELECT * 
FROM eventos_por_usuario_dia 
WHERE usuario_id = 'USR1001'; 
Esta consulta no debería ser válida porque no se está usando la clave de partición 
completa. La clave de partición está compuesta por: 
usuario_id + fecha 
Por eso Cassandra espera recibir ambos valores. 
Conclusión del paso 
Cuando una clave de partición está compuesta por más de una columna, la consulta 
debe incluir todas esas columnas para acceder eficientemente a la partición.

16. Eliminar un registro 
Para eliminar un registro también se necesita su clave. 
Ejemplo: 
DELETE FROM eventos_por_usuario_dia 
WHERE usuario_id = 'USR1001' 
AND fecha = '2026-05-03' 
AND evento_ts = '2026-05-03 08:15:00' 
AND evento_id = 00000000-0000-0000-0000-000000000000; 
Después consultar nuevamente: 
SELECT * 
FROM eventos_por_usuario_dia 
WHERE usuario_id = 'USR1001' 
AND fecha = '2026-05-03'; 
Observación 
En Cassandra, las eliminaciones tienen una lógica interna particular, porque los datos 
se marcan inicialmente como eliminados mediante tombstones y se eliminan 
físicamente más adelante durante procesos internos de compactación. Para esta 
práctica alcanza con comprender el efecto visible del DELETE.

Actividad de cierre 
Como vimos en esta práctica, Cassandra no se utiliza como una base relacional 
tradicional. Aunque CQL usa una sintaxis parecida a SQL, el diseño depende de las 
consultas que se deben responder. 
El aprendizaje principal es: 
Consulta esperada → clave de partición → clave de clustering → tabla optimizada 
Una buena tabla Cassandra no es necesariamente la más normalizada ni la más 
general. Es aquella que permite recuperar los datos requeridos de forma eficiente, con 
una clave bien diseñada y evitando búsquedas innecesarias.
